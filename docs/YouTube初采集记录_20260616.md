# YouTube 初采集记录 — 2026-06-16

> 项目：UNESCO 中国非遗在 YouTube / TikTok 跨平台传播研究  
> 本轮目标：用 YouTube Data API 跑通 44 项、61 个可用搜索词的 US baseline 浅采样，并在拿到 3 个 key 后扩到每词 3 页，为后续相关性清洗、深采与 YouTube vs TikTok 对齐做准备。  
> 当前主口径：`regionCode=US`、`order=relevance`、不设置 `relevanceLanguage`。

## 二、采集脚本

新增/更新脚本：

- `scripts/collect_youtube_search.py`

功能：

1. 读取 `config/youtube_ich_search_terms.v1.json`。
2. 调用 YouTube Data API v3：
   - `search.list`：按关键词分页采视频 ID 与基础 snippet。
   - `videos.list`：批量补齐 `viewCount`、`likeCount`、`commentCount`、`tags`、`duration` 等字段。
3. 输出 term 级 meta 与 video 级数据。
4. 记录 quota 估算：
   - `search.list` = 100 quota / page
   - `videos.list` = 1 quota / 50 ids
5. 区分状态：
   - `sample_limit_reached`：本轮浅采页数后还有下一页；这是人为采样上限，不是 YouTube 500 条平台天花板。
   - `ceiling_capped_500`：未来满采 10 页后仍有下一页，才标为 500 上限触顶。
   - `exhausted`：自然采空，无下一页。
   - `zero_real`：非失败条件下 0 条。
   - `failed`：API/网络/配额等技术失败，不能当作 0 结果。
6. 已支持多 API key：
   - `--api-key-file` 可读取多行 key。
   - search 按 term 轮换 key。
   - 如果某个 key/API 调用失败，会尝试下一个 key。
   - meta 记录 `api_key_index`、`api_key_attempts`、`search_calls_including_failed_attempts`。

密钥处理：

- 单 key 本地文件：`.youtube_api_key`
- 多 key 本地文件：`.youtube_api_keys`
- `.gitignore` 已加入 `.youtube_api_key*`
- 不提交密钥。

## 五、字段对齐 TikTok 的当前设计

视频级主要字段：

- `video_id`
- `url`
- `search_keyword`
- `project_id`
- `heritage_item`
- `title`
- `description`
- `tags`
- `publishDate`
- `viewCount`
- `likeCount`
- `commentCount`
- `channelId`
- `channelTitle`
- `duration`
- `region_code`
- `search_order`

对齐关系：

- YouTube `viewCount` ≈ TikTok `playCount`
- YouTube `likeCount` ≈ TikTok `diggCount`
- YouTube `commentCount` ≈ TikTok `commentCount`
- YouTube `search_keyword` ≈ TikTok `source_term`
- YouTube term-level `stop_reason` ≈ TikTok term-level depth/ceiling verdict，但 YouTube 当前 1–3 页浅采只能叫 `sample_limit_reached`，不能叫真正触顶。

## 六、初步观察（只作采集质量观察，不作研究结论）

1. 3 页扩采后仍有 56/61 个词有下一页，说明多数词还未触及 YouTube 可见上限。
2. 5 个词在 3 页内自然结束，但 `totalResults_estimate` 对其中一些词仍给出很大数字，证明 `totalResults` 不可信作精确存量。
3. YouTube 对泛词/英文词召回量很大，后续相关性清洗很关键；尤其是：
   - `Li textile`
   - `Hua'er`
   - `seal engraving`
   - `Chinese wooden movable type`
   - `Xi'an drum music`
4. `viewCount` 字段基本可用，适合先做 YouTube 项目级触达汇总。
5. 目前没有 `quotaExceeded` 终止；3 key 足够支撑下一步继续采，但 key#1 在中后段出现过失败，不能假设单 key 稳定。

## 七、下一步建议

### A. 先做 YouTube 初采 relevance 打标 / 项目级触达汇总

基于 3 页扩采的 8,463 行，先派生：

- `data/derived/youtube_initial_20260616_pages3/youtube_project_reach_summary.csv/json`
- `data/derived/youtube_initial_20260616_pages3/youtube_video_relevance_labels.ndjson`

用 TikTok 的相关性口径 v1 平移，但需要适配 YouTube：

- title + description + tags 作为匹配文本。
- negative terms 来自 YouTube 表与 TikTok config 两套来源合并。
- 不删除噪声，只标 `likely_relevant / needs_review / low_relevance`。

### B. 再决定是否深采到 500

已知总可用词 K=61：

- 1 页/词：约 6,156 quota（已完成）
- 3 页/词：约 18,860 quota（已完成，含失败重试）
- 10 页/词（500 条/词）：理论约 61,610 quota；考虑失败重试，实际应预留 65,000+ quota

如果继续满采 500/词：

- 3 key 理论日配额约 30,000 quota/day
- 全量约 2–3 天
- 但建议先看 3 页样本的相关性/触达分布，再决定是否所有词都值得满采。

### C. 深采策略

建议不要马上全词 500。更稳的节奏：

1. 当前 3 页扩采：已完成。
2. 先做相关性标签和项目级触达汇总。
3. 第三轮只对重点/高价值/决策敏感词采到 500：
   - YouTube 高触达候选。
   - TikTok P0/P1 对照项目。
   - `stock_high × reach_low` 与 `stock_low × reach_high` 这类关键象限。
   - 前 3 页发现高 viewCount 且相关性可能成立的词。

## 八、不要踩的坑

1. 不把 `sample_limit_reached` 误写成平台 500 触顶。
2. 不把 `totalResults_estimate` 当精确存量。
3. 不把 API `failed/quotaExceeded` 混入 `zero_real`。
4. 不用江茜表里的「量感(待核实)」作判断。
5. 不在 git 里提交 `.youtube_api_key`、`.youtube_api_keys` 或任何 API key。
6. 不用 raw viewCount 直接做最终跨平台结论；先做相关性清洗。
