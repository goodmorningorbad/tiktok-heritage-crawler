## 二、采集脚本

新增脚本：

- `scripts/collect_youtube_search.py`

功能：

1. 读取 `config/youtube_ich_search_terms.v1.json`。
2. 调用 YouTube Data API v3：
   - `search.list`：按关键词分页采视频 ID 与基础 snippet。
   - `videos.list`：批量补齐 `viewCount`、`likeCount`、`commentCount`、`tags`、`duration` 等字段。
3. 输出 term 级 meta 与 video 级数据。
4. 记录 quota 估算：
   - `search.list` = 100 quota / page
   - `videos.list` = 1 quota / 50 ids
5. 区分状态：
   - `sample_limit_reached`：本轮浅采 1 页后还有下一页；这是人为采样上限，不是 YouTube 500 条平台天花板。
   - `ceiling_capped_500`：未来满采 10 页后仍有下一页，才标为 500 上限触顶。
   - `exhausted`：自然采空，无下一页。
   - `zero_real`：非失败条件下 0 条。
   - `failed`：API/网络/配额等技术失败，不能当作 0 结果。

密钥处理：

- API key 写入本地 `.youtube_api_key`。
- `.youtube_api_key` 已加入 `.gitignore`。
- 不提交密钥。

## 六、初步观察（只作采集质量观察，不作研究结论）

1. 61 个词里 60 个 1 页后仍有下一页，说明第一轮浅采未触及平台可见上限，只是完成摸底。
2. 很多词的 `totalResults_estimate` 返回 1,000,000，明显是估算/截断式信号，不能当精确存量。
3. YouTube 对泛词/英文词召回量很大，后续相关性清洗会很关键；尤其是：
   - `Li textile`
   - `Hua'er`
   - `seal engraving`
   - `Chinese wooden movable type`
   - `Xi'an drum music`
   这类词需要人工/规则核查噪声。
4. 采集字段和 API 流程已跑通，可进入第二阶段规划：全词 2–3 页扩采，或对重点词深采到 500。

## 七、下一步建议

### A. 先做 YouTube 初采 relevance 打标 / 项目级触达汇总

基于已采 3,027 行，先派生：

- `data/derived/youtube_initial_20260616/youtube_project_reach_summary.csv/json`
- `data/derived/youtube_initial_20260616/youtube_video_relevance_labels.ndjson`

用 TikTok 的相关性口径 v1 平移，但需要适配 YouTube：

- title + description + tags 作为匹配文本。
- negative terms 来自 YouTube 表与 TikTok config 两套来源合并。
- 不删除噪声，只标 `likely_relevant / needs_review / low_relevance`。

### B. 再决定配额扩采节奏

已知总可用词 K=61：

- 1 页/词：约 6,156 quota（已完成）
- 2 页/词：约 12,322 quota，单 key 约 2 天
- 3 页/词：约 18,483 quota，单 key 约 2 天
- 10 页/词（500 条/词）：约 61,610 quota，单 key 约 7 天

如果有 3 个 Google key：

- 理论日配额约 30,000 quota/day
- 全量 500/词约 2–3 天可完成
- 但脚本需要加 key pool / quota failure rotation，不能把 key 写死。

### C. 深采策略

建议不要马上全词 500。更稳的节奏：

1. 本轮 1 页浅采：已完成。
2. 第二轮全词扩到 2–3 页：看前 100–150 条的相关性与触达分布。
3. 第三轮只对重点/高价值/决策敏感词采到 500：
   - YouTube 高触达候选。
   - TikTok P0/P1 对照项目。
   - `stock_high × reach_low` 与 `stock_low × reach_high` 这类关键象限。
   - 第一轮发现高 viewCount 且相关性可能成立的词。

## 八、不要踩的坑

1. 不把 `sample_limit_reached` 误写成平台 500 触顶。
2. 不把 `totalResults_estimate` 当精确存量。
3. 不把 API `failed/quotaExceeded` 混入 `zero_real`。
4. 不用江茜表里的「量感(待核实)」作判断。
5. 不在 git 里提交 `.youtube_api_key` 或任何 API key。
6. 不用 raw viewCount 直接做最终跨平台结论；先做相关性清洗。
