## 当前结论

- search 通道拿不到可靠的平台总结果数，只能记录采样行数、是否触顶、停止时 `has_more` 是否仍为 true。
- hashtag 通道可通过 TikTok `/api/challenge/detail/` 拿到 `challengeInfo.stats.videoCount/viewCount`，但只能解释为“标签总规模（含噪声）”，不等于该非遗传播量。
- hashtag 和 search 必须分开；hashtag 不做 search fallback。
- 泛词不进入 hashtag_terms 建议；频次高不是保留理由。判定标准：tag 单独拿出来是否指向具体非遗。
- 冷门项目不凑数。修复 tag 后仍采不到，就如实记录采不到，低存在感本身是结论线索。

### 3. 还没做正式 search 摸底

口径已定：先摸底再校准 N。

- 若超过半数项目触顶：上调 N，或头部主要用 hashtag `videoCount/viewCount` 分层。
- 若绝大多数内容极少：低端分档更细，例如 0 / 几条 / 几十条 / 百条级。
- 触顶项目使用 `sample_rows_total + search_hit_cap_terms + search_has_more_at_stop` 解读为 `>=N`。

### 4. 还没做账号/IP pool runner

现在只是记录来源元数据，还没有自动轮换账号/IP。

正式摸底前建议加一个 runner：

- 输入账号/IP 池。
- 每个项目或每几个项目固定分配一个账号/IP。
- 同一轮摸底保持统一 region。
- assignment 写入 report，避免混采不可追溯。

建议最低：3 个中性账号 + 3 条同 region 住宅 IP；更稳：5 个中性账号 + 5 条同 region 住宅 IP。

### 5. 还没做降质/地域偏置核对

摸底跑完后，挑热门/中等/冷门项目各若干，用干净号/IP 手动核对平台实际量级，与采集结果对比。

目的：确认 TikTok 没有静默风控降质，例如少给结果、折叠播放量、地域偏置严重。

## 不要踩的坑

- 不要把 search fallback 塞进 hashtag 通道。
- 不要用 hashtag `videoCount/viewCount` 直接代表非遗传播量。
- 不要用固定样本噪声率衡量出海程度；噪声率主要反映词义歧义。
- 不要给冷门项目加宽泛 tag 凑采集量。
- 不要混多个 region 后再做一个总排名，除非明确是在做 region 混采，且每条数据都有 region 标记。
- 长期搜索中国非遗的账号会形成偏好，可能系统性高估相关内容；采集账号应尽量中性，并记录 `collector_account_role`。
